# DESIGN

В документе приведены некоторые принятые решения, возможные направления развития проекта и моменты, которые можно обсудить.

## Формат файла

В заголовке файла реализованно хранение только количества записей.

В заголовок есть смысл добавить:

- сигнатуру файла;
- версию формата;
- контрольную сумму (хеш) данных.

Для дальнейшего уменьшения размера файлов можно использовать различные методы сжатия. Это в свою очередь может повысить производительность программы, если бутылочным горлышком была работа с диском.

## Плотность хранения данных

Битовые поля кодируются в один общий байт.
Запись в файл делается с помощью "ручной" сериализации (убираем паддинг). Уменьшение размера каждой записи:
- сэкономит дисковое пространство;
- повысит производительность  программы, за счёт уменьшения работы с диском.

`#pragma pack` не используется, так как платформо-зависима.

## JoinDump()

Для объединения массивов используется хеш-таблица с линейным пробированием.

В качестве хеш-функции используется МурМур.

## LoadDump()

Память под массив выделяется внутри `LoadDump()`, а **ответственность за её освобождение полностью лежит на вызывающей стороне.**

## FillBuffer()

Код упаковки записи в бинарный формат выделен в отдельную функцию.

В дальнейшем её можно использовать например, при передаче данных по сети, или записи в другой объект.

## SortDump()

Для сортировки используется стандартная функция `qsort()`.

В рамках задания применение чего-то вроде интрасорт избыточно.
Так как структуры имеют небольшой размер, локальность хорошая, то сортировка индексов не применяется.

## Компаратор

Компаратор сделан так чтобы сортировка была детерминированной.
В рамках тестового задания это избыточно, и несёт незначильный накладной расход, но это позволит избежать различий для разных реализаций `qsort()`.

Сравнение чисел с плавающей точкой выполняется по "мягкому сравнению". Для данного задания было бы достаточно строгого сравнения, однако мягкое сравнение делает функцию более пригодной для дальнейшего использования.

## Стресс-тест

Данный сценарий не требовался спецификацией, но для проверки работоспособности на максимальных размерах входных данных был реализован стресс-тест.

Для обеспечения воспроизводимости, для генерации тестовых данных использовался build-in генератор с фиксированным начальным состоянием.

## Ограничения

Тестовая утилита предполагает запуск из каталога рядом с основной программой. При необходимости поддержки запуска из произвольно места можно внести изменения в обработку аргументов.

### Дополнительные варианты оптимизации

При увеличении размеров обрабатываемых файлов и ужесточении требований к производительностя возможно:

- использование многопоточного объединения и сортировки;
- SIMD;
- использование специализированных сортировок (например: файлы начнут превышать размеры доступной памяти; или изменятся свойства входных данных);
- сжатие данных

## Тестовое покрытие

Проверяются:

- корректность слияния массивов;
- суммирование полей `count` и `cost`;
- выбор значений полей `primary` и `mode`;
- объединение записей с повторяющимися `id` в одном и разных массивах;
- устранение дублирующихся `id`;
- сортировка результирующего массива по полю `cost`;
- работа с пустыми массивами;
- работоспособность при больших объёмах данных.
